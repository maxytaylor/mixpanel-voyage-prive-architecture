01 — Background

Why architecture matters here

The POC has demonstrated that Mixpanel can power the product analytics use cases Voyage Privé needs — funnel analysis, segmentation, PM self-serve, retention. The data is there. The question being raised now is the right one: what does a production-ready, long-term architecture look like, and where does Mixpanel sit within it?

Vincent's instinct — that BigQuery and Snowplow should remain the source of truth, with Mixpanel as the analysis layer reading from them — is the right framing. This document proposes how to get there, and in what order.

→
Core principle.

BigQuery is the source of truth. Mixpanel is the lens. Any fix, enrichment or new property is made once in BigQuery and Mirror propagates it automatically — no re-instrumentation, no dual maintenance.

02 — Current State

Three structural problems with today's setup

The POC architecture is functional — the data we've been analysing proves it. But it's not production-ready as a long-term foundation. Three issues need to be resolved before building on top of it.

Problem 01

The data layer hasn't been audited

Snowplow is enriching and forwarding events, but missing context, unnested array columns, undefined UTMs and gaps in business properties all suggest the instrumentation predates any product analytics intent. Messy data in, messy data in.

Problem 02

The ingestion journey is too long

GTM → Snowplow → BigQuery → Warehouse Connector means multiple failure points and meaningful latency. Mirror runs daily, so Mixpanel is always ~24 hours behind live product behaviour — fine for analysis, but worth understanding as a constraint.

Problem 03

No mobile app instrumentation

The mobile app has no event tracking today. 78% of search events and a significant share of product views are either missing entirely or arriving via a partial mobile-web view. Mobile-native behaviour is invisible.

⚠
The data layer audit is the critical path.

Until the BigQuery view produces clean, flat, correctly-typed data with proper identity mapping, any other architectural change just moves the same problems faster or to a different location. The audit comes first.

03 — Target Architecture

The recommended end state

The proposed architecture keeps Snowplow and BigQuery as the collection and enrichment backbone, with Mixpanel reading from BigQuery via the Warehouse Connector in Mirror mode. No new pipeline complexity until the foundation is solid.

The key principle is that no enrichment or data modelling happens inside Mixpanel. Mixpanel receives clean, pre-modelled data from BigQuery. Any fix is made once in the BigQuery view, and Mirror reflects it automatically — including historically. This is the Mirror advantage that directly addresses VP's goal of one source of truth.

04 — Ingestion Path

Two paths: choose your timeline

Vincent's suggestion of forwarding events from Snowplow directly to Mixpanel post-enrichment is architecturally valid and Mixpanel supports it natively. But the right choice depends on when the data layer audit is complete. Here's the honest comparison:

OPTION A Recommended now

BigQuery → Warehouse Connector (Mirror)

Continue the current architecture. BigQuery is master — all enrichment, flattening, identity resolution and cleaning happens there. Mixpanel reads from BigQuery via Mirror. BigQuery also serves BI tools, data science and any future analytics consumers.

No new pipeline work required Mirror handles retroactive fixes automatically BigQuery remains the audit trail and compliance record Single change propagates to all consumers Simpler to govern — one data model to maintain Daily sync — Mixpanel ~24h behind live behaviour Not suitable for real-time alerting use cases
OPTION B Consider post-audit

Snowplow → Mixpanel HTTP forwarding (real-time)

Configure Snowplow to forward enriched events directly to Mixpanel's ingestion API in near real-time, bypassing the daily sync cycle. Mixpanel has a native Snowplow integration that supports this pattern.

Real-time dashboards and alerting Removes daily sync latency entirely Supported natively via Mixpanel's Snowplow integration Requires Snowplow pipeline configuration change Data messy upstream = data messy in Mixpanel, faster Creates two consumers of Snowplow output to manage Historical backfill still requires BigQuery anyway
→
The decision rule.

If the primary goal is one source of truth and clean data, Option A is correct today. Option B becomes worth evaluating once the data layer audit is complete and the BigQuery view is producing reliable output. Running Option B on top of messy data just delivers bad data faster.

A note on Google Tag Manager

GTM as a collection mechanism introduces a layer of indirection that is hard to audit and version-control. It fires events based on DOM interactions, which is inherently fragile as the product evolves. Two considerations worth raising:

If the majority of data gaps stem from GTM configuration issues, this is an argument for migrating toward server-side tracking — sending events from the application layer directly to Snowplow's collector endpoint. If GTM is working correctly and the gaps are downstream (enrichment, flattening in BigQuery), it can stay as-is.

The data layer audit should make this clear. Either outcome is fine — what matters is that the decision is made deliberately, not by default.

05 — Mobile Strategy

Closing the mobile gap

No mobile app instrumentation today is a material blind spot — 78% of search events on the platform come from Phone users, and mobile converts at a fraction of desktop. Without native app tracking, the journey of the majority of VP's users is partially invisible.

Three options in order of recommended priority, each with a different implementation cost:

Option 1 — Recommended

Snowplow Mobile SDK (iOS / Android)

Add Snowplow's native mobile tracker to the app. Events enrich and land in the same BigQuery table, then flow to Mixpanel via the existing Warehouse Connector. Keeps BigQuery as the single source of truth and maintains the architecture cleanly. Most implementation effort, but the right long-term answer.

Option 2 — Fast start

Mixpanel iOS / Android SDK

Instrument the native app directly and send events to Mixpanel in parallel with web. Fast to deploy, but creates a second ingestion path that diverges from BigQuery. Useful to get mobile data quickly while the Snowplow mobile implementation is planned.

Option 3 — Interim only

Mixpanel Autocapture (mobile web)

Add Mixpanel's JS SDK with autocapture to the mobile web experience. No native app changes needed. Very limited in scope — can't capture in-app interactions — but captures mobile web sessions quickly and at zero instrumentation cost.

→
Pragmatic path.

Deploy Option 3 now to capture mobile web sessions immediately. Run Option 1 (Snowplow mobile) as a planned project. This gives VP mobile insight within days while the native instrumentation is properly scoped and built.

07 — Governance Model

One source of truth requires active governance

The architecture makes BigQuery the source of truth technically. Governance makes it the source of truth organisationally. Without it, different teams will build different definitions of the same metric, trust in the data erodes, and PMs revert to asking the data team for SQL. This is the outcome VP is trying to avoid.

BigQuery layer (data team owns)

  • Canonical SQL view — reviewed and version-controlled
  • Approved column list — no unmapped arrays or raw Snowplow fields
  • Change process — any schema change goes through review before sync
  • Identity model — user_id + device_id mapping documented and owned
  • Bot filtering — is_bot column maintained and applied

Mixpanel Lexicon layer (data owner + PMs)

  • All core events verified and named in plain language
  • Properties described — no raw Snowplow names visible to PMs
  • Tags applied by domain: Booking, Search, Marketing, Identity
  • Raw / technical properties hidden from the default property picker
  • New events go through Event Approval before appearing in reports
→
Mixpanel's hub-and-spoke governance model fits VP's structure.

A central data owner (Mathieu / Ahmed's team) owns the BigQuery view and Lexicon standards. Product team leads (Vincent's team) own their domain's events and dashboards. PMs are spoke consumers — they explore and build within governed, verified data, without needing SQL.
